**Sections:** §12.7, §14.3
**Ambiguity:** The spec says batch operations MUST NOT roll back, MUST continue processing, and MUST report per-file results. But the test format doesn't define how to simulate I/O errors during batch execution. This is needed to test partial failure behavior distinct from validation failure (which aborts the entire batch before execution).
**Assumption:** Tests use `simulate.io_error_on: "path"` in the batch operation input to inject an I/O error when writing a specific file during the execution phase. The test runner must fail the write for that file while allowing others to succeed.
**Resolution:** §14.3.1 "Extended Test Format" added to 14-conformance.md, documenting `simulate` as an extended input field (covers `io_error_on` alongside `external_modify` and `external_create`).
