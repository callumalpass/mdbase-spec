**Sections:** §12.1, §6.3, §6.4
**Issue:** When the create operation is given an explicit type (e.g., `type: "person"`), the spec does not require that the written file satisfy that type's match rules. This means a create operation can produce a file that is immediately orphaned from its type — it was scaffolded using the type's schema, defaults, and generated fields, but cannot be identified as that type by any subsequent operation.
This is a real problem when `explicit_type_keys: []`. The type input is used for schema lookup during creation, but no `type` key is written to frontmatter. If the type's match rules depend on a field value (e.g., `where: {tags: {contains: "person"}}`), the file is only matchable if that field is present — which depends on whether the field has a default and whether defaults are materialized. If the match rule depends on `path_glob`, the file is only matchable if it was created at a conforming path. None of these dependencies are enforced.
**Proposed resolution:** §12.1 should add a step after writing: "When the create operation is given an explicit type, the written file MUST satisfy that type's match rules (§6.3–6.4). If the file cannot satisfy the match rules given the provided inputs, defaults, and generated values, the create operation MUST fail with an error rather than produce an unmatchable file."
This applies specifically to create with an explicit type. Update operations do not specify a type and should not be constrained — a user may intentionally modify match-relevant fields to reclassify a file.
**Implementation notes:**
- For `where`-based match rules (e.g., `tags: {contains: "person"}`): the effective frontmatter (including defaults) must satisfy the `where` clause. If a required tag has no default and the user didn't provide it, create should fail.
- For `path_glob`-based match rules: the final file path (explicit or derived from `path_pattern`) must match the glob. If the user provides a path outside the glob, create should fail or warn.
- For `fields_present`-based match rules: the effective frontmatter must contain the required fields. Null values from unresolved generated fields do not satisfy `fields_present`.
**Conformance tests needed:**
1. Type with `where: {tags: {contains: "x"}}` and `tags: {default: [x]}` — create succeeds, file is matchable.
2. Type with `where: {tags: {contains: "x"}}` and no default for `tags`, user does not provide `tags` — create should fail (unmatchable).
3. Type with `path_glob: "people/*.md"`, user provides path `notes/bob.md` — create should fail or warn (path doesn't match glob).
4. Type with `fields_present: [date]`, user does not provide `date` and no default — create should fail.
5. Type with `explicit_type_keys: []` and `where`-based matching — create does not insert `type` key, file is still matchable via match rules.

**Resolution:** §12.1 now requires explicit-type creates to satisfy match rules when match evaluation is supported and introduces `match_failed`. Added Level 2 tests in `tests/level-2/create-match-rules.yaml` covering `where`, `path_glob`, and `fields_present` enforcement.
