**Sections:** §7.4, §7.5
**Severity:** Medium (interoperability risk)
**Issue:** The spec defines `integer` and `number` types with `min`/`max` constraints but never specifies the valid range for the values themselves. Key questions left open:
1. What is the maximum/minimum integer value an implementation must support? JavaScript's `Number.MAX_SAFE_INTEGER` (2^53 - 1)? 64-bit signed integers? Arbitrary precision?
2. What is the precision of `number` values? IEEE 754 double? Arbitrary precision decimal?
3. What happens when a YAML file contains an integer that overflows the implementation's native type (e.g., a 128-bit integer in a language using 64-bit ints)?

Without guidance, implementations in different languages will silently disagree on whether large values are valid. A YAML file with `count: 9999999999999999999` might work in Python (arbitrary precision) but silently lose precision in JavaScript.
**Proposed resolution:** Add a minimum required range to §7.4 and §7.5:
- `integer`: Implementations MUST support at least the signed 64-bit range (−2^63 to 2^63 − 1). Values outside this range MAY produce `constraint_violation`.
- `number`: Implementations MUST support at least IEEE 754 double precision. Implementations MAY support higher precision.
- Add a note that YAML parsers may silently truncate large integers and implementations SHOULD detect this.
**Resolution:** §7.4 now requires at least signed 53-bit integer range; §7.5 requires IEEE 754 double precision.
