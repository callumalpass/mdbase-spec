**File:** `tests/level-2/matching-eval.yaml`
**Sections:** §6.4, §5.12
**Bug:** Commit `39bf1a6` ("Add chapter frontmatter, type definitions, timezone setting, and reorganize conformance tests") inserted a new "Group X: Match rules cannot reference computed fields" between Group 3's tests, accidentally capturing two unrelated tests into the new group.

The tests "custom plural type key works" and "default type key is not recognized when custom keys configured" were originally part of the "custom explicit_type_keys" group (Group 3), which provides `explicit_type_keys: [kind, kinds]` in its config. After the reorganization, they are nested under the "match rule referencing computed field is invalid" group, which has a different config (no `explicit_type_keys`) and a `bad.md` type that correctly fails validation due to a computed field in `match.where`.

As a result, Collection::open fails for these two tests (because the bad type is rejected), and they can never pass.

**Affected tests:**
1. "custom plural type key works" -- expects `types: [task, note]`, gets `None` (collection fails to open)
2. "default type key is not recognized when custom keys configured" -- expects `types: [note]`, gets `None`

**Proposed fix:** Move both tests back into Group 3 ("custom explicit_type_keys") so they inherit the correct setup with `explicit_type_keys: [kind, kinds]`.
