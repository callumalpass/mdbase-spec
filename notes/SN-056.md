**Sections:** ¬ß7.15, ¬ß5.6, ¬ß10.5
**Issue:** The `generated` field strategy supports deriving values from other fields (`{from, transform}`), and `filename_pattern` derives filenames from field values ‚Äî but the reverse direction is not supported. There is no way to derive field values from file metadata. Rather than a special-purpose `from_filename` mechanism, file metadata should be available as source variables in the existing `{from, transform}` generated field system.
**Use cases:**
1. Importing existing files that already follow a naming convention (e.g., daily notes named `2024-03-15.md` should auto-populate a `date` field)
2. Keeping a `slug` field in sync with the filename stem without manual duplication
3. Migration from untyped collections where the filename *is* the structured data
4. Deriving category or grouping fields from directory structure
**Proposed variables:** Expose file metadata as `from` sources using a `file.` prefix, mirroring the file properties already defined in ¬ß10.5:

| Variable | Description | Example (for `notes/2024/meeting-notes.md`) |
|----------|-------------|----------------------------------------------|
| `file.name` | Filename stem (no extension) | `meeting-notes` |
| `file.basename` | Full filename with extension | `meeting-notes.md` |
| `file.ext` | File extension | `md` |
| `file.path` | Path relative to collection root | `notes/2024/meeting-notes.md` |
| `file.folder` | Parent directory relative to root | `notes/2024` |

**Proposed syntax:**
```yaml
fields:
  slug:
    type: string
    generated:
      from: file.name           # "meeting-notes"

  category:
    type: string
    generated:
      from: file.folder         # "notes/2024"
      transform: slugify

  date:
    type: date
    generated:
      from: file.name           # would need a parse/extract transform
```
**Consistency:** These variable names align with the `file.*` properties already queryable via ¬ß10.5, keeping the mental model unified ‚Äî the same names mean the same things whether you're querying or generating.
**Open questions:**
1. Is `{from, transform}` expressive enough, or does extracting structured data from filenames (e.g., a date from `2024-03-15-title.md`) require a `parse` or `pattern` option beyond simple transforms?
2. Should `file.*` variables also be available in `filename_pattern` itself (e.g., `filename_pattern: "{file.folder}/{id}.md"` to preserve directory structure)?
3. Should these be treated as generated-on-create only (like `ulid`), or recomputed on read (since the filename could change via rename)?

**Resolution:** `generated: {from, transform}` now accepts `file.*` metadata sources (`file.name`, `file.basename`, `file.ext`, `file.path`, `file.folder`) per ¬ß10.5. Added tests in `tests/level-1/field-types-gaps.yaml`. `file.*`-derived fields are computed from the final resolved path and MUST NOT be referenced by `path_pattern` to avoid cycles.
