**Sections:** §12.1, §5.6, §7.15
**Issue:** §12.1 specifies the create operation steps as: (1) determine types, (2) apply defaults, (3) generate values, (4) validate, (5) determine path. Step 5 says: "If type has `filename_pattern`, derive from field values." The ordering implies that generated values (step 3) are available when the path is derived (step 5), but this is not stated explicitly. An implementation bug was found where the path was derived from the **input** frontmatter (user-provided values only) rather than the **effective** frontmatter (which includes generated fields and defaults). This caused `filename_pattern: "{zettelid}.md"` to produce the literal filename `zettelid.md` instead of using the generated ULID, because the `zettelid` field had not yet been populated at path-derivation time.
**Spec clarification needed:** §12.1 step 5 should explicitly state: "Implementations MUST resolve `filename_pattern` template variables against the effective frontmatter, which includes generated field values (step 3) and applied defaults (step 2)." The numbered step ordering already implies this, but the dependency must be made explicit to prevent implementations from resolving the pattern too early.
**Conformance gap:** No existing test exercises `filename_pattern` with a generated field variable during create. Need tests in level-1 that verify:
1. `filename_pattern: "{id}.md"` where `id` has `generated: ulid` — create without explicit path should produce a ULID-based filename, not `id.md`.
2. `filename_pattern: "{date}-{slug}.md"` where `date` has `generated: now` and `slug` is derived `{from: title, transform: slugify}` — create should use the generated/derived values, not literal key names.
3. `filename_pattern: "{id}.md"` where `id` has `generated: ulid` and the user provides an explicit `path` — the explicit path should take precedence over the pattern (existing behavior, but should be tested alongside the above).
**Impact:** Without this clarification and tests, implementations can silently produce garbage filenames for any `filename_pattern` that references generated fields — the most common use case for the feature.

**Resolution:** §12.1 now requires `path_pattern` (formerly `filename_pattern`) to resolve against the effective frontmatter (defaults + generated). Added Level 1 tests in `tests/level-1/operations.yaml` for generated/default-based path derivation and alias coverage.
