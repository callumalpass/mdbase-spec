**Sections:** §5
**Proposal:** Type definitions should support an optional key (e.g., `display_name_key`) that specifies which field to use as the human-readable display name for records of that type. Different types naturally use different fields as their label — tasks have `title`, people have `name`, meetings might have `subject`. Without a spec-level convention, tools must guess which field to display.
**Proposed syntax:**
```yaml
name: task
display_name_key: title
fields:
  title:
    type: string
    required: true
  status:
    type: enum
    values: [open, done]
```
**Open questions:**
1. Should the referenced field be required to exist in the type's `fields` definition?
2. Should it be constrained to fields that resolve to a string, or allow any field (with implicit toString)?
3. Fallback behavior when the key is absent from the type definition, or the referenced field is missing/empty on a specific record — filename (sans extension) is the obvious fallback.
4. Should computed fields be valid targets?

**Resolution:** Added optional `display_name_key` to type schema. Implementations derive `file.display_name` from that field (effective value) and fall back to `file.basename` when missing/empty. Added Level 3 tests in `tests/level-3/file-metadata-and-context-gaps.yaml`.
