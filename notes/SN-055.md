**Sections:** §7.15
**Issue:** The current `generated` strategies (`ulid`, `uuid`) produce fixed-length identifiers (26 and 36 characters respectively). For small or single-user collections, these are often longer than necessary, producing unwieldy filenames. A configurable-length random alphanumeric string strategy would fill the gap between full global-uniqueness guarantees and practical short identifiers.
**Proposed syntax:**
```yaml
fields:
  id:
    type: string
    generated:
      random: 8   # 8-character alphanumeric string
```
**Collision handling:** Not a concern for the generation strategy itself. If uniqueness is required, the existing `unique: true` field constraint (§7.2) already provides cross-file duplicate detection. Implementations that enforce `unique: true` would detect collisions at validation time. Whether to retry on collision is an implementation concern, not a spec concern.
**Open questions:**
1. Should the alphabet be configurable (e.g., `alphanumeric`, `hex`, `base32`), or is alphanumeric sufficient?
2. Should there be a minimum length to discourage obviously collision-prone configurations?

**Resolution:** Added `{random: N}` generated strategy for string fields with alphanumeric `a-z0-9` alphabet and length 1–64. Invalid configuration is `invalid_type_definition`. Added tests in `tests/level-1/field-types-gaps.yaml` (generation) and `tests/level-1/types-basic.yaml` (invalid length).
