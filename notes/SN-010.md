**Sections:** §6.2
**Ambiguity:** When `settings.explicit_type_keys` is set to e.g. `[kind, kinds]`, it's not explicitly stated whether the default keys `type` and `types` still work alongside the custom keys, or are completely replaced.
**Assumption:** Custom `explicit_type_keys` completely replaces the defaults. If you set `[kind, kinds]`, then `type` and `types` become ordinary fields with no special meaning. This is consistent with the spec's note: "If you want to use a field like `type` as ordinary data, remove it from `settings.explicit_type_keys`".
**Resolution:** §6.2 now states that custom keys replace defaults. Existing strict-mode and explicit-type-key tests already cover this behavior.
