**Sections:** §5.5, §5.8
**Severity:** Low (internal consistency)
**Issue:** §5.5 defines `strict` as accepting three value types: `false` (boolean), `true` (boolean), and `"warn"` (string). The meta type definition in §5.8 (line 365–367) models this as:
```yaml
strict:
  type: enum
  values: ["true", "false", "warn"]
```
This is technically correct because YAML boolean `true` coerces to string `"true"` via §7.16 type coercion. However, it means the meta type's own schema doesn't accurately represent the actual type of the `strict` field — it's modeling a boolean/string union as a string enum. A type definition file with `strict: true` (YAML boolean) passes validation only because of implicit coercion, not because the schema correctly describes the field.

SN-059 already notes this design choice ("frontmatter booleans (`strict: true`) coerce to strings via §7.16"). The observation here is that this is a pragmatic workaround for the lack of a union type in the type system, not an accurate schema.
**Proposed resolution:** No spec change needed — the coercion path works. Add a comment to the meta type definition in §5.8 explaining the rationale: "The `strict` field accepts YAML booleans `true`/`false` and string `"warn"`. It is modeled as a string enum because the type system does not support union types; boolean values are coerced to strings via §7.16."
**Resolution:** Added the explanatory note in §5.8.
