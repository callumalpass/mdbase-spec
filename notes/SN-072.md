**Sections:** §5.2, §5.13
**Severity:** Medium (normative inconsistency)
**Issue:** The `display_name_key` fallback behavior uses different RFC 2119 keywords in two places:
- §5.2 (line 111, in the type definition schema example): "If missing or empty on a record, implementations SHOULD fall back to file.basename"
- §5.13 (line 481, the dedicated section): "If the field is missing or empty on a record, implementations MUST fall back to `file.basename`"

§5.13 is the authoritative section for this feature, so MUST is likely the intended keyword. The SHOULD in §5.2 is in a code comment within a YAML example, which makes it easy to overlook — but code comments in spec examples are still normative unless marked otherwise.
**Proposed resolution:** Change the §5.2 comment to MUST: `# If missing or empty on a record, implementations MUST fall back to file.basename`. Alternatively, remove the RFC 2119 keyword from the example comment entirely and rely on §5.13 as the single authoritative statement.
**Resolution:** Updated the §5.2 comment to MUST to match §5.13.
