**Sections:** §5.9, §14.3
**Ambiguity:** §5.9 says "Implementations MUST provide a way to create type definition files" and describes the expected steps (validate schema, check name conflicts, write file, reload registry). However, the conformance test format in §14.3 doesn't define a `create_type` operation. The test format only lists standard operations (validate, read, create, update, delete, rename, query, etc.) for content files, not type definition files.
**Assumption:** Tests use `operation: create_type` with structured input (`name`, `fields`, `parent`, `strict`). Implementations translate this into their native type creation API. The operation writes a type definition file and reloads the type registry. Errors use the same error codes as type loading (`invalid_type_definition`, `missing_parent_type`, `path_conflict`).
**Resolution:** §14.3.1 "Extended Test Format" added to 14-conformance.md, documenting `create_type` as an extended operation.
