**Sections:** §12.1
**Issue:** §12.1 shows an output example that includes a `path` field, but does not state that the output MUST include `path`. When the path is derived from `filename_pattern` (i.e., the caller did not provide an explicit path), the derived path is the only way the caller knows what file was created. Without a MUST requirement, an implementation could omit the path from the output, leaving CLIs and API consumers unable to report what happened.
**Proposed resolution:** §12.1 output section should state: "The output MUST include the `path` field containing the final file path relative to the collection root. This is especially important when the path was derived from `filename_pattern` rather than provided by the caller."
**Conformance tests needed:**
1. Create with explicit path — output includes `path` matching the input.
2. Create with derived path (via `filename_pattern`) — output includes `path` with the resolved filename.
3. Create with derived path — the `path` in the output matches the file actually written to disk.

**Resolution:** §12.1 now requires `path` in create output. Existing and new tests in `tests/level-1/operations.yaml` assert returned paths for explicit and derived cases.
