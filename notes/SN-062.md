**Sections:** ¬ß4, ¬ß5.11, ¬ß12.1
**Issue:** ¬ß12.1 step 7 says implementations "SHOULD omit fields that were filled solely by defaults, unless the caller explicitly requests default materialization." The escape hatch assumes a programmatic API where each call can pass options, but in practice most usage is through CLIs where per-invocation flags are tedious. More importantly, the current default (omit) causes a mismatch between what the create operation reports (effective frontmatter including defaults) and what's actually on disk. This is confusing: the CLI shows fields like `title`, `givenName`, `familyName` in the output, but the file doesn't contain them.
**Proposed setting:**
```yaml
settings:
  write_defaults: true   # default: true
```
When `true`, create and update operations write default-only fields to disk. When `false`, the current omission behavior is preserved.
**Why `true` should be the default:**
1. **Least surprise.** The output of a create operation should match what's on disk. Users expect that if the CLI says a field exists with a value, it's in the file.
2. **Self-contained files.** Files are portable and readable without needing the type definition to infer missing fields. This matters for files viewed in other editors, shared across tools, or moved between collections.
3. **Explicit over implicit.** A file that says `title: ""` communicates "this field exists and is empty." A file that omits `title` requires the reader to know the schema to understand the full state.
4. **Schema evolution is overrated as a default.** The argument for omitting defaults is that changing a default in the type definition propagates to existing files. In practice, silently changing the effective value of existing files is risky ‚Äî if you change a default from `""` to `"Unknown"`, you probably want to review existing files rather than have them silently change meaning. Opting into that behavior with `write_defaults: false` is the safer direction.
5. **Collection-level consistency.** Multiple tools operating on the same collection should agree on whether defaults are materialized. A per-call option leads to inconsistent files depending on which tool last touched them.
**Spec changes needed:**
1. ¬ß4 ‚Äî Add `write_defaults` to the settings schema with default `true`.
2. ¬ß12.1 step 7 ‚Äî Change SHOULD omit to: "If `settings.write_defaults` is `false`, implementations SHOULD omit fields filled solely by defaults. If `true` (the default), implementations MUST write all fields including those filled by defaults."
3. ¬ß5.11 ‚Äî Update schema evolution guidance to note that `write_defaults: false` enables automatic default propagation, with the tradeoff of files not being self-contained.
**Conformance tests needed:**
1. Create with `write_defaults: true` (default) ‚Äî default-only fields appear in the written file.
2. Create with `write_defaults: false` ‚Äî default-only fields are omitted from the written file but present in the effective output.
3. Update with `write_defaults: true` ‚Äî defaults are materialized if not already present.
4. Read with omitted defaults ‚Äî effective frontmatter includes defaults regardless of the setting (the setting only affects writes).

**Resolution:** Added `settings.write_defaults` (default `true`) to configuration, updated ¬ß12.1/¬ß5.11 behavior, and added Level 1 tests in `tests/level-1/operations-gaps.yaml`. Existing default-suppression tests were updated to set `write_defaults: false`.
