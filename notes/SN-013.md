**Sections:** §11.18, §11.4
**Ambiguity:** The spec lists `type_error` for "Cannot add string and number" but also says "type mismatch returns null and emits type_error". In a `where` clause, should `"hello" + 5` silently return null (and thus not match), or should it abort the query with an error? The difference matters: if it returns null, the query continues; if it's an error, the query fails.
**Assumption:** In a `where` filter expression, `"hello" + 5` produces a query-level `type_error` error because it's a static type mistake (not a data-dependent issue like null access). This is different from runtime null propagation. The test expects the query to fail with `type_error`.
**Resolution:** The original assumption was **incorrect**. Per §11.18, type mismatch errors "MUST NOT abort the overall query" — they return null. A note added to §11.18 clarifying that error-handling rules apply to all expression contexts including `where` filters. The test updated to expect 0 results (null → file excluded) instead of a `type_error` abort.
