**Sections:** §6.3, §6.4, §12.3, §7.15
**Issue:** The spec says type matching uses both explicit type keys (§6.2) and match rules including `path_glob` (§6.3–6.4). §12.3 (update) says implementations determine types for the file and apply generated field strategies. However, all existing conformance tests for update + generated fields use explicit type keys in the frontmatter (e.g., `type: task`). No test exercises update with types assigned solely via `match_rules` / `path_glob`. This allowed an implementation bug where the update operation resolved types from frontmatter only (ignoring the file path), causing `generated: now_on_write` fields to silently not trigger for path-matched types.
**Conformance gap:** Need tests in level-2 that combine path-based type matching with update operations and `generated: now_on_write`. Specifically:
1. Type with `match: { path_glob: "..." }` and a `now_on_write` field, no explicit type key in the file — update should trigger the generated field.
2. Type matched by `path_glob` that `extends` a parent with a `now_on_write` field — update should trigger the inherited generated field.
3. Type matched by `fields_present` — update should also resolve types correctly.
**Impact:** Without these tests, implementations can pass the full conformance suite while silently failing to apply generated fields for any type not assigned via explicit frontmatter keys.

**Resolution:** §12.3 now specifies type determination using explicit keys **and** match rules (including path-based). Added Level 2 tests in `tests/level-2/update-type-matching.yaml` covering `path_glob`, `fields_present`, and inherited `now_on_write` behavior.
