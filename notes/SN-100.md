**Sections:** §5.2, §7.1, §14.7
**Severity:** High (cross-implementation interoperability)

**Issue:** The spec allows extensions in general (§14.7), but does not define behavior for unknown keys inside field definitions (`fields.<name>.*`).

Today this creates portability risk:
- One implementation may keep unknown field-definition keys and expose them to extension consumers.
- Another may parse into a fixed struct and silently drop unknown keys.

Both behaviors can still appear "spec-compliant" because no section currently defines ignore/preserve semantics for field-definition extension metadata.

This blocks a practical pattern used by tools that need semantic mapping metadata in schemas (for example, app-specific field-role annotations).

**Proposed resolution:** Add explicit extension-key rules for field definitions.

1. **Ignore unknown keys:** Implementations MUST NOT reject type definitions solely because a field definition (including nested object/list item definitions) contains unknown keys.
2. **Preserve unknown keys:** Implementations SHOULD preserve unknown keys in any type-inspection/load API output and in any operation that reads and rewrites type definitions.
3. **No semantic effect by default:** Unknown keys MUST have no core-spec semantic effect unless explicitly standardized.
4. **Namespace recommendation:** Non-standard keys SHOULD be namespaced to avoid collisions (for example `tn_role`, `app_x`, or `vendor.foo`).

**Conformance gap:** Add tests that load types containing unknown field-definition keys at:
- top-level fields,
- `list.items`, and
- `object.fields.*`
and assert they are accepted. If the conformance surface includes a type-inspection output, assert keys are preserved there as well.

**Suggested wording target:** §7.1 "Field Definition Structure" is the most direct place, with a cross-reference from §14.7.
