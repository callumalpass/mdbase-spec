**Sections:** §5.6, §12.1
**Issue:** §5.6 says `filename_pattern` uses `{}` to reference field values, but does not define what happens when a template variable cannot be resolved. There are three distinct failure cases:
1. The referenced field exists in the schema but has no value (null/undefined) — e.g., a `generated` field that yielded null because its source was missing.
2. The referenced field does not exist in the schema at all — e.g., a typo like `{titel}`.
3. The referenced field has a value that produces an invalid filename — e.g., a string containing path separators or null bytes.

The reference implementation has inconsistent behavior across codepaths: the create operation falls back to the literal key name (producing filenames like `zettelid.md`), while the validation codepath uses an empty string (producing filenames like `.md`). Neither is desirable.
**Proposed resolution:** Implementations MUST return an error when a `filename_pattern` variable cannot be resolved to a non-empty string value. Specifically:
- If a variable references a field with no value (null, undefined, or empty string), the create operation MUST fail with an error rather than producing a malformed filename.
- If a variable references a field not defined in the type schema, type loading SHOULD emit a warning.
- The error code should be `path_required` (already defined in Appendix C) with a message indicating which variable could not be resolved.

Silently producing garbage filenames is always worse than failing with a clear error. The literal-key-name fallback and empty-string fallback are both bugs, not features.
**Conformance tests needed:**
1. `filename_pattern: "{id}.md"` where `id` has no value and no `generated` strategy — create without explicit path should fail with `path_required`, not produce `id.md`.
2. `filename_pattern: "{id}.md"` where `id` is `generated: {from: title, transform: slugify}` and `title` is not provided — create should fail (derived field yields null), not produce a broken filename.
3. `filename_pattern: "{missing}.md"` where `missing` is not a field in the schema — type loading should warn, create should fail.

**Resolution:** §5.6 now defines unresolvable variables as `path_required`, and type loading warns on unknown variables. Added tests in `tests/level-1/error-code-hardening.yaml` and `tests/level-1/types-basic.yaml`.
