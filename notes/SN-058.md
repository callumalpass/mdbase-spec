**Sections:** §7.15
**Issue:** The current `generated` strategies produce either globally unique identifiers (`ulid`, `uuid`), timestamps (`now`, `now_on_write`), or derived values (`{from, transform}`). There is no strategy for producing sequential integer identifiers. Sequences are a common pattern for issue trackers, numbered notes, invoices, and any collection where human-readable ordering matters.
**Proposed syntax:**
```yaml
fields:
  number:
    type: integer
    generated: sequence
```
**Semantics:** On create, scan all existing files of the matched type(s), find the maximum value of this field, and assign max + 1. If no files exist or no file has a value for this field, start at 1.
**Scope:** The sequence is per-type by default — each type maintains its own counter independently. A `task` type and a `bug` type would each have their own sequence.
**Open questions:**
1. Should the starting value be configurable (e.g., `sequence: { start: 100 }`)?
2. Should the scope be configurable (per-type vs per-collection)?
3. Gap handling: if file #3 is deleted, should the next file be #4 (no gap reuse) or #3 (fill gaps)? No-gap-reuse (max + 1) is simpler and avoids surprising renumbering.
4. Performance: scanning all files on every create could be slow for large collections. Implementations may cache the current max value and invalidate on delete/rename.
5. Should this work with `unique: true` automatically, or must the user specify both?

**Resolution:** Added `sequence` generated strategy for integer fields with defaults: start=1, scope=per-type, no gap reuse. Optional config supports `start` and `scope`. Added Level 1 tests in `tests/level-1/field-types-gaps.yaml` and validation for invalid usage in `tests/level-1/types-basic.yaml`.
