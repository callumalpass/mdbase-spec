id: SN-009
title: "`where exists: true/false` vs `fields_present` semantics"
sections:
  - "§6.4"
status: resolved
kind: ambiguity
---

**Sections:** §6.4
**Ambiguity:** The spec says `fields_present` requires the key to exist AND be non-null. The `where` `exists` operator says "Field is present (true) or missing (false)". It's not explicit whether `where: { assignee: { exists: true } }` has the same null-handling semantics as `fields_present: [assignee]` — i.e., does `exists: true` consider a null-valued field as "not existing"?
**Assumption:** `where: { field: { exists: true } }` behaves the same as including the field in `fields_present` — the field must be present AND non-null. This keeps the two mechanisms consistent.
**Resolution:** §6.4 updated. The `exists` operator description changed to "Field is present and non-null (true) or missing/null (false)." A note added after the `where` operators table confirming `exists: true` uses the same semantics as `fields_present`.
