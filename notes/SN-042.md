**Sections:** §7.11, §7.16
**Issue:** L1 spec-coverage-gaps.yaml tests that lists containing integers/booleans with `items: { type: string }` are coerced to strings (per §7.16 "string accepts any scalar via toString"). L2 matching-eval.yaml had a `list_item_invalid` test using `tags: [valid, 123, true]` with string items type, which contradicted L1 by expecting rejection of coercible values.
**Resolution:** Changed the L2 test to use `scores: [5, "not-a-number", 3]` with integer items type. A non-numeric string genuinely cannot be coerced to an integer, making the `list_item_invalid` expectation correct without contradicting L1 coercion rules.
