**Sections:** §5.8, §12, Appendix A.5
**Proposal:** When a collection is initialized (e.g., `mdbase init`), implementations should create a `meta` type in `_types/` alongside the `mdbase.yaml` configuration file. The meta type describes the schema for type definition files themselves — it matches files in `_types/` and defines all possible type-definition fields.

**Purposes:**
1. **Self-description**: The meta type documents every field available in type definitions, in-collection
2. **Validation**: Type definition files can be validated against the meta type during loading
3. **Bootstrapping**: New types can be created as records of the meta type, using the same create operation used for any other record

**Proposed `_types/meta.md`:**
```markdown
---
name: meta
description: Schema for type definition files

match:
  path_glob: "_types/**/*.md"

strict: false

fields:
  name:
    type: string
    required: true
    pattern: "^[a-z][a-z0-9_-]{0,63}$"
    description: Type name. Must match filename without extension.

  description:
    type: string
    description: Human-readable description of the type.

  extends:
    type: string
    description: Parent type name for single inheritance.

  strict:
    type: enum
    values: ["true", "false", "warn"]
    default: "false"
    description: "Unknown field handling: false (allow), true (reject), or \"warn\"."

  match:
    type: object
    fields:
      path_glob:
        type: string
        description: Glob pattern for path-based matching.
      fields_present:
        type: list
        items:
          type: string
        description: Fields that must be present for a match.
      where:
        type: any
        description: Field predicates using expression operators.
    description: Rules for automatic file-to-type association.

  filename_pattern:
    type: string
    description: "Pattern for deriving filenames from fields using {field} placeholders."

  fields:
    type: any
    description: >
      Field definitions mapping. Keys are field names, values are field
      definition objects. See spec section 7 for the complete field
      definition schema including type-specific constraints.
---

# Meta

The meta type describes the structure of type definition files. It is created
automatically during collection initialization and matches all files in the
types folder.

**Type-Level Fields**

### Required

- **name**: Must match the filename without `.md`. Lowercase letters, numbers,
  hyphens, and underscores only. Max 64 characters.

### Optional

- **description**: What this type represents
- **extends**: Parent type to inherit fields from (single inheritance only)
- **strict**: How to handle unknown fields — `true`, `false`, or `"warn"`
- **match**: Rules for automatically matching files to this type
- **filename_pattern**: Pattern for generating filenames using `{field}` placeholders
- **fields**: A mapping of field name to field definition

**Field Definition Options**

Each entry in `fields` supports:

| Option | Type | Description |
|--------|------|-------------|
| `type` | string | **Required.** `string`, `integer`, `number`, `boolean`, `date`, `datetime`, `time`, `enum`, `list`, `object`, `link`, `any` |
| `required` | boolean | Must the field be present? Default: `false` |
| `default` | any | Default value when field is missing |
| `generated` | string/object | `ulid`, `uuid`, `now`, `now_on_write`, or `{from, transform}` |
| `description` | string | Human-readable field description |
| `deprecated` | boolean | Mark field as deprecated |
| `unique` | boolean | Cross-file uniqueness constraint |
| `computed` | string | Expression for computed fields (Level 3) |

Plus type-specific constraints (`min_length`, `max_length`, `pattern`, `min`,
`max`, `values`, `items`, `fields`, `target`, `validate_exists`, etc.).
```

**Design notes:**
- The meta type uses `strict: false` because type definitions contain fields (like type-specific constraints within field definitions) that cannot be fully enumerated by the meta type
- The `fields` field uses type `any` because field definitions are polymorphic — available options depend on the field's `type` value, and field names are user-defined
- The `strict` field uses type `enum` with string values `["true", "false", "warn"]` — frontmatter booleans (`strict: true`) coerce to strings via §7.16
- The meta type matches itself via `path_glob: "_types/**/*.md"`, which is intentional
- §2.3 excludes `_types/` from the regular file scan but type files are still scanned separately for type loading — the meta type's match rule applies during that separate scan

**Spec changes needed:**
1. §5.8 (currently "Built-in vs User Types" which says "no built-in types") — update to introduce the meta type as a bootstrapping type created by init
2. §12 or Appendix A.5 — formalize what `init` creates (currently only a CLI example)
3. Appendix A collection structures — add `meta.md` to `_types/` listings

**Open questions:**
1. Should the meta type be MUST (required by init) or SHOULD (recommended)? If missing/deleted, implementations should still function.
2. Should `mdbase type create` use the meta type's schema for validation, or continue using its own internal validation as described in §5.9?
3. The `init` operation itself needs a formal definition — currently it only appears as a CLI example in Appendix A.5. Should it become a named operation in §12?

**Resolution:** Added a required meta type created by `init` (§5.8), formalized `init` in §12.12, and updated Appendix A examples. Added Level 1 tests in `tests/level-1/init.yaml`.
