---
id: SN-061
title: "Field override in inheritance silently strips `generated` and other metadata"
sections:
  - "§5.4"
  - "§7.15"
status: resolved
kind: conformance_gap
---

**Sections:** §5.4, §7.15
**Issue:** §5.4 rule 3 states: "Field override: Child fields with the same name override parent fields completely." This is correct and tested (spec-coverage-gaps.yaml tests that a child can remove `required` by redefining a field without it). However, the "completely" replacement semantics create a subtle footgun when a child type redefines a field without restating all parent properties. Consider:
```yaml
# Parent (base.md)
fields:
  zettelid:
    type: string
    generated: ulid

# Child (person.md) — intends to just declare the field exists
extends: base
fields:
  zettelid:
    type: string   # silently strips generated: ulid
```
The child's `zettelid: {type: string}` completely replaces the parent's `zettelid: {type: string, generated: ulid}`. The `generated` strategy is silently lost. This is particularly dangerous because:
1. The child author may not realize the parent had `generated` on the field — they just listed the field for documentation purposes.
2. There is no warning or error when a child override strips a `generated` strategy.
3. The resulting failure is indirect: `filename_pattern: "{zettelid}.md"` silently produces bad filenames, or the field is unexpectedly `null`.
**Spec clarification needed:** §5.4 should add a note after rule 3 warning that complete replacement includes all field properties (`generated`, `required`, `default`, `unique`, `deprecated`, constraints). If a child intends to narrow only one property (e.g., change `max` from 3 to 5), it MUST restate all other parent properties it wishes to preserve. Implementations SHOULD consider emitting a warning when a child override on a field strips a `generated` strategy from the parent.
**Conformance gap:** Need tests that verify:
1. Child redefines a `generated: ulid` field with just `type: string` — the generated strategy is NOT inherited (current correct behavior, but untested).
2. Child redefines a `generated: ulid` field and restates `generated: ulid` — the strategy is preserved.
3. Child redefines a field with `generated: now` on parent, sets `generated: uuid` — child's strategy wins.
4. (Optional, if warning is adopted) Child strips `generated` from parent field — implementation emits a warning.
**Impact:** Without documentation of this behavior, type authors will commonly redefine parent fields for documentation or minor changes and unknowingly break generated field strategies. Combined with SN-060, this produces a double failure: the field value is missing AND the filename pattern falls back to literal key names.

**Resolution:** §5.4 now warns that field overrides replace all properties (including `generated`) and recommends warnings when a child strips a generated strategy. Added Level 1 tests in `tests/level-1/types-basic.yaml` covering generated override removal, preservation, and replacement.
