**Sections:** §5.6
**Issue:** The spec uses `filename_pattern` (§5.6), which implies the pattern only covers the filename component (e.g., `my-note.md`). In practice, types often need to control the directory structure too — for example, `people/{slug}.md`, `diary/{date}.md`, or `posts/{year}/{slug}.md`. The reference implementation already accepts full relative paths in the pattern and internally maps `filename_pattern` to `path_pattern`, but the spec doesn't acknowledge this.
**Options:**
1. **Rename to `path_pattern`** — Accurately describes the feature's scope. Accept `filename_pattern` as a deprecated alias for backward compatibility.
2. **Keep `filename_pattern` but clarify** — State that the value is a relative path from the collection root, not just a filename. The name becomes misleading but avoids a breaking change.
3. **Add a separate `path_pattern`** — `filename_pattern` controls the filename only, `path_pattern` controls the full relative path. More expressive but adds complexity.

Option 1 is the cleanest. The current name actively misleads type authors into thinking directory prefixes aren't supported.
**Spec changes needed:**
1. §5.6 — Rename to `path_pattern` (or clarify scope). Update all examples.
2. §12.1 step 5 — Update reference from `filename_pattern` to the new name.
3. §14 (conformance test format) — Update any references.
4. If renaming, add a note that `filename_pattern` is accepted as an alias for backward compatibility.
**Conformance tests needed:**
1. `path_pattern: "people/{slug}.md"` — create derives path including directory, directory is created automatically.
2. `path_pattern: "{year}/{slug}.md"` with `year` derived from a date field — create produces nested directory structure.
3. (If aliased) `filename_pattern` still works as an alias for `path_pattern`.

**Resolution:** Renamed spec field to `path_pattern` with `filename_pattern` as a deprecated alias. Updated examples and added alias coverage in `tests/level-1/operations.yaml`.
