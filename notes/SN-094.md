**File:** `tests/level-3/query-namespaces.yaml`
**Sections:** §10.5, §11.3
**Bug:** In Group 1 ("note namespace accesses raw persisted frontmatter"), the test setup defines two files:

```yaml
tasks/a.md:
  type: task
  title: "Explicit status"
  status: done

tasks/b.md:
  type: task
  title: "Default status"
```

The type `task` defines `priority` as an integer field with `default: 3`. Neither file includes an explicit `priority` value in its raw frontmatter.

Two tests depend on `tasks/a.md` having an explicit `priority` value:

1. "note.* returns null for field filled only by default" (`where: "note.priority == null"`) -- expects only `tasks/b.md` to match, but both files lack `priority` in raw frontmatter, so both have `note.priority == null` and both match.
2. "bare access returns effective value (with default applied)" (`where: "priority == 3"`) -- expects only `tasks/b.md` to match, but both files receive `priority: 3` from defaults, so both match.

**Affected tests:**
1. "note.* returns null for field filled only by default" -- expects 1 result (`tasks/b.md`), gets 2 (`tasks/a.md`, `tasks/b.md`)
2. "bare access returns effective value (with default applied)" -- expects 1 result (`tasks/b.md`), gets 2 (`tasks/a.md`, `tasks/b.md`)

**Proposed fix:** Add an explicit `priority` value to `tasks/a.md`, e.g.:

```yaml
tasks/a.md: |
  ---
  type: task
  title: "Explicit status"
  status: done
  priority: 5
  ---
```

This would make the first test pass (only `b.md` has `note.priority == null`) and the second test pass (only `b.md` has effective `priority == 3`).
