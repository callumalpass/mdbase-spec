**Sections:** Various
**Issue:** The spec does not define:
1. Floating-point precision expectations in expressions (e.g., `0.1 + 0.2`).
2. Valid date ranges for date/datetime types (min/max years).
3. Symlink handling during scans (follow vs ignore, root-escape prevention).
4. UTF-8 BOM handling before `---` frontmatter delimiter.
5. YAML keyword field names (`yes`, `no`, `null`) in type definitions.
6. Whether `mdbase.yml` is acceptable or only `mdbase.yaml`.
7. Explicit rejection of empty type names (currently implicit).
8. Match rules referencing computed fields (type assignment circularity).
9. Object field nesting depth limits in type schemas.
10. Multiple `---` blocks (only first is frontmatter?).
11. Watcher behavior during batch operations (intermediate events vs final state).
12. `write_nulls: omit` with `required: true` on update (ordering of validation vs write).
13. Concurrency semantics for `generated: sequence` (duplicate prevention).
**Proposed resolution:** Add a short "Edge Cases" subsection to §14.6 with guidance, and add targeted tests for the most divergent items (3, 4, 8, 10, 13).
**Resolution:** Addressed key edge cases directly in relevant sections (BOM handling, multiple frontmatter blocks, symlink scanning, date ranges, computed/match constraints, directory creation, etc.). Remaining low-impact items are documented as implementation guidance.
