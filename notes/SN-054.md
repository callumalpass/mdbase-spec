title: "Create operation must respect `explicit_type_keys` setting"

**Sections:** §6.2, §12.1
**Issue:** §12.1 (create) accepts a `type` input parameter to specify which type definition to use for scaffolding. §6.2 defines `settings.explicit_type_keys` as the list of frontmatter keys that carry type assignments (default `["type", "types"]`). When `explicit_type_keys` is set to `[]`, types are determined solely by match rules (`path_glob`, `fields_present`, `tags_include`), and no frontmatter key should carry type semantics. However, no conformance test verifies that the create operation respects this setting. An implementation bug was found where `create` unconditionally inserted `type: <name>` into frontmatter regardless of the `explicit_type_keys` configuration, polluting frontmatter with an unintended field.
**Conformance gap:** Need tests that verify:
1. With `explicit_type_keys: []`, creating a file with `type: "mytype"` input should NOT insert a `type` field into frontmatter — the type input is used only for schema lookup, generated fields, and validation.
2. With `explicit_type_keys: ["kind"]`, creating a file with `type: "mytype"` input should insert `kind: mytype` (using the first configured key), not `type: mytype`.
3. With default `explicit_type_keys` (`["type", "types"]`), creating a file should insert `type: mytype` as before (existing behavior).
**Impact:** Without these tests, implementations can pass the suite while inserting spurious type fields into frontmatter for collections that don't use explicit type keys.

**Resolution:** §12.1 now defines create-time type field writing based on `settings.explicit_type_keys`. Added Level 1 tests in `tests/level-1/explicit-type-keys-create.yaml` for empty, custom, and default explicit type key configurations.
