# mdbase v0.3.0 operator guide

## Version model

- Collection protocol: `0.3.0`
- Runtime profile: `0.1.0`
- Package versions: independent SemVer versions

Prerelease packages may implement the final `0.3.0` protocol. New collections
write `spec_version: "0.3.0"`. TypeScript and Rust retain an explicit read-only
compatibility path for collections created with the earlier
`0.3.0-alpha.1` marker.

## Upgrade a collection

1. Stop tools that may write to the collection.
2. Work from a copied collection or a clean version-control checkpoint.
3. Install the v0.3 runtime/core/CLI versions in publication order.
4. Produce and review a non-mutating report:

```bash
mdbase migrate v0.3 analyze --report migration-report.json
```

5. Resolve every `unsupported`, `invalid_records`, and `target_diagnostics`
   entry. Do not use partial migration merely to suppress a design mismatch.
6. Apply the exact reviewed report:

```bash
mdbase migrate v0.3 apply --report migration-report.json --yes
```

7. Record the reported backup path, then validate the collection and its
   priority consumers before allowing writes again:

```bash
mdbase validate --format json
```

The apply phase verifies source hashes before writing, creates a durable backup
and journal, validates staged output, and validates the post-write collection.

## Recover a migration

Stop all collection writers. Use the backup path returned by apply:

```bash
mdbase migrate v0.3 recover \
  --backup .mdbase/migrations/v0.3-<id> \
  --yes
```

Recovery verifies the journal and current migrated hashes before replacing
files. After recovery, reinstall the prior consumer versions and validate the
restored v0.2 collection. Do not delete the backup until the migrated collection
has passed its full application smoke tests.

## Author a provider

Import generic types from the browser-safe package, not from TaskNotes or a
platform adapter:

```ts
import type {
  MdbaseRuntimeProvider,
  MdbaseRuntimeEventEnvelope,
} from "@callumalpass/mdbase-runtime";
```

A provider must return a canonical descriptor, return every advertised
contract, report readiness, implement subscriptions and dispatch for contracts
it owns, and dispose its resources. Registration is atomic: malformed schemas,
ownership conflicts, an unavailable readiness result, or an advertised contract
mismatch leave the effective registry unchanged.

The platform adapter selects and owns the host. In Obsidian that adapter is
`mdbase-obsidian`; provider plugins discover its API by plugin ID. They must not
use TaskNotes as a generic service locator.

Before exposing an effectful action, define its input/output JSON Schemas,
capability requirements, effects, and provider ownership. Exercise registration,
preflight, denied policy, successful dispatch, invalid input/output, event
delivery, deduplication, unregister, and disposal behavior.

## Author a workflow

Workflow records use the canonical `workflow` type and include stable IDs for
the workflow, triggers, and steps. Each trigger references a registered event;
each step references a registered action. Dynamic values use the v0.3 CEL
expression object:

```yaml
input:
  path:
    $expr: event.payload.file.path
```

Run preflight before enabling a workflow. Preflight must resolve provider
version ranges, events, actions, capabilities, embedded schemas, and the
selected executor policy. Keep workflows disabled until a dry run or copied-data
test proves their effects. The runtime contract package validates and dispatches
contracts; workflow scheduling and execution belong to an executor such as
TaskNotes Workflows or another platform runtime.

TaskNotes Workflows 0.1.x files that use `tasknotes-workflow`, `schemaVersion`,
trigger or step `type`, and camelCase run fields are compatibility inputs rather
than canonical workflow records. The plugin does not rewrite them at startup.
Use its **Migrate workflow files to the mdbase runtime format** command to
review per-file diffs and invalid files. Apply rechecks source hashes, creates a
backup and manifest under `.obsidian/tasknotes-workflows/workflow-migrations/`,
preserves Markdown bodies, and restores files
already changed if a later write fails. Saving one legacy workflow through the
visual editor is also an explicit canonical conversion.

The checked `examples/v0.3/canvas-runtime` collection is the smallest complete
provider/event/action/policy/workflow example.

## Publish the ecosystem

1. Tag the exact promoted spec, schemas, fixtures, claims, and runtime profile.
2. Publish `@callumalpass/mdbase-runtime`.
3. Replace the TypeScript core's local runtime link with the exact published
   version, regenerate its lockfile, run the full suite and tarball smoke, then
   publish `@callumalpass/mdbase`.
4. Replace the CLI and other core consumers' local links with the published core
   version, regenerate locks, rerun qualification, and publish them.
5. Publish the `mdbase` Rust crate, then package and publish `mdbase-lsp` and
   dependent Rust applications against the registry crate.
6. Publish `@tasknotes/model`, publish `mdbase-tasknotes` against the registry
   core and model packages, then release `mdbase-obsidian`, TaskNotes, TaskNotes
   Workflows, and Canvas Bases.
7. Validate copied downstream collections before declaring the ecosystem release
   complete.

No dependent release may contain a sibling `file:` dependency or a path-only
Cargo dependency. Registry-resolved lockfiles and installed-tarball/crate tests
are evidence produced during this sequence, not before the prerequisite package
exists.
