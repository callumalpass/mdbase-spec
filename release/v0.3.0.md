# mdbase v0.3.0 Ecosystem Release Contract

## Status

This is the auditable release contract for the stable mdbase `0.3.0` ecosystem.
It distinguishes required release behavior from useful follow-up work.

Current implementation version: `0.3.0-alpha.1`.

Runtime profile version: `0.1.0`.

Stable `0.3.0` is not complete until every required gate below has evidence.

Local implementation qualification is complete and the protocol has been
promoted to the repository root. The next package version should be a release
candidate, not stable. Stable remains blocked on multi-platform CI, registry
install verification in publication order, and live Obsidian smoke evidence.

## Ownership Decisions

### Collection Protocol

`mdbase-spec` owns the normative collection protocol, canonical schemas,
profile identifiers, conformance fixtures, and migration rules. Implementations
consume those artifacts; they do not redefine them.

### TypeScript Runtime Package

Generic runtime contracts and the reference in-memory provider host belong in a
small browser-safe package in the mdbase package family. The target package name
is `@callumalpass/mdbase-runtime`, versioned with runtime profile `0.1.x`.

The package owns:

- canonical runtime types and schemas
- provider registration and removal
- deterministic contract composition
- preflight and capability-policy checks
- event-envelope validation and delivery
- action input/output validation and dispatch
- platform-neutral host lifecycle

Node-specific Markdown loading and materialization, if shipped in the same
package, MUST live behind an explicit `./node` export. Importing the main package
MUST NOT load Node APIs, SQLite, Obsidian APIs, or TaskNotes code.

### Platform Host Discovery

The protocol does not prescribe global process or plugin discovery. Platform
adapters inject or broker one selected host instance.

For Obsidian, the `mdbase-obsidian` plugin is the selected host adapter. It
exposes the host through its plugin API and owns host disposal with the vault
lifecycle. Independent provider and executor plugins discover that API by the
`mdbase-obsidian` plugin ID. They MUST NOT use TaskNotes as the generic service
locator. A temporary TaskNotes compatibility host MAY exist during alpha
migration, but it MUST be removed before the release candidate.

### Providers And Executors

TaskNotes and Canvas Bases are optional providers. Neither owns the generic
runtime contract. TaskNotes Workflows is an optional executor and may remain a
TaskNotes-focused product, but generic event/action preflight and dispatch use
the mdbase runtime package.

Canvas Bases MUST remain fully usable without TaskNotes. TaskNotes-specific
actions are available only when the `tasknotes` provider is registered and
passes preflight.

## Stable Release Gates

### Specification

- [x] No unresolved normative questions in chapters 00-16.
- [x] Every canonical schema validates as Draft 2020-12.
- [x] Conformance claims validate against the canonical claim schema.
- [x] Examples and migration fixtures are generated or checked reproducibly.
- [x] The promoted site and archived v0.2 documentation both build.

### Runtime Ownership

- [x] The generic TypeScript runtime package is browser-safe and publishable.
- [x] Its main export carries immutable or isolated canonical schemas and the
      reference host enforces them for live providers and event envelopes.
- [x] TaskNotes model, TaskNotes Workflows, and Canvas Bases use its canonical
      types rather than duplicate generic interfaces.
- [x] The reference host implementation no longer lives in TaskNotes.
- [x] `mdbase-obsidian` exposes the selected host and provider discovery works
      without TaskNotes installed.
- [x] Canvas Bases emits `canvas.drop` and runs generic workflows without
      TaskNotes.
- [x] TaskNotes actions appear and dispatch only when its provider is available.
- [x] TaskNotes Workflows writes canonical `workflow/0.1` records, registers
      them with the selected host, and keeps legacy files read-only until an
      explicit backed-up migration is approved.

### Data Safety And Compatibility

- [x] v0.2 collections remain readable by transition-capable tools.
- [x] Migration analysis is non-mutating and reproducible.
- [x] Apply requires explicit approval and verifies source hashes.
- [x] Backups, rollback, interruption, stale-report, invalid-record, and partial
      migration behavior have fault-injection coverage.
- [x] TaskNotes, Pickle, and one non-Obsidian collection complete migration
      rehearsals from copied real-world data.

### Implementations

- [x] TypeScript and Rust include validated conformance claim documents.
- [x] A second implementation passes shared `core_read` and
      `collection_semantics` fixtures.
- [x] LSP diagnostics preserve canonical codes, paths, fields, and schema
      locations.
- [x] Unsupported profiles and operational limits are explicit.

### Release Qualification

- [ ] Linux, macOS, and Windows CI pass for publishable packages.
- [ ] Fresh tarball/crate installs pass without sibling links.
- [x] Production and development dependency audits have no known high or
      critical findings.
- [ ] Live Obsidian smoke tests pass for TaskNotes, Workflows, and Canvas Bases.
- [x] Performance baselines pass on representative small and large collections.
- [x] Upgrade, rollback, provider-authoring, and workflow-authoring documentation
      is complete.

## Publication Order

1. Tag the exact spec, schemas, fixtures, and runtime profile artifacts.
2. Publish `@callumalpass/mdbase-runtime`.
3. Publish `@callumalpass/mdbase`.
4. Publish `mdbase-cli` and the Rust crates/binaries.
5. Publish `@tasknotes/model` against the runtime package, then publish
   `mdbase-tasknotes` against the registry core and model packages.
6. Release `mdbase-obsidian` as the selected Obsidian host adapter.
7. Release TaskNotes, then TaskNotes Workflows and Canvas Bases.
8. Validate downstream collections before declaring the ecosystem release
   complete.

Every dependent lockfile MUST resolve the published prerelease or stable
artifact, not a sibling link or an older compatibility branch, before its
release is cut.

The exact upgrade, recovery, provider, workflow, and staged dependency commands
are in [v0.3.0-operator-guide.md](./v0.3.0-operator-guide.md).

## Prerelease Progression

- Alpha releases may change schemas and package boundaries.
- Beta begins only after normative schemas and public exports freeze.
- Release candidate begins only after required consumers use the generic host.
- Stable `0.3.0` requires every gate in this document to have durable evidence.
