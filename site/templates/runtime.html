Runtime profile 0.1

Portable contracts, host-owned execution

Collections can declare events, actions, workflows, capabilities, and policy as typed records. Platform hosts register live providers, validate contracts, authorize effects, and execute only the behavior they explicitly support.

Read runtime contracts Read workflows
Ownership

The contract is portable; the host owns effects

01

Providers advertise typed surfaces

A provider descriptor names the action, event, capability, and workflow contracts it can actually serve.

02

Policy is default-deny

The selected platform host validates inputs and requires explicit capability policy before effectful dispatch.

03

Materialization stays optional

Implicit provider contracts can remain in memory or be written as Markdown for inspection without creating behavior by themselves.

Canonical example

A Canvas drop becomes a validated record patch

These records come from the checked `examples/v0.3/canvas-runtime` fixture used by the TypeScript and Rust contract tests.

providers/canvas-bases.md

type: provider

Host flow

Registration and dispatch fail closed

Provider registration is atomic. The host validates the descriptor, advertised contracts, embedded JSON Schemas, readiness, ownership, and selected policy before contracts become visible.

Package boundary

One browser-safe contract package

`@callumalpass/mdbase-runtime` exposes canonical schemas, provider types, and the reference in-memory host from its main export. Markdown loading and materialization are isolated behind `@callumalpass/mdbase-runtime/node`. Neither entry point depends on TaskNotes, and the browser entry loads no Node or Obsidian APIs.
